## Обзор

Логика отмены активной брони клиента. Применяется на экране деталей брони (SCR-009) и регламентирует полный флоу: проверку доступности отмены по временному порогу, модальное подтверждение, вызов API отмены, обработку ответов и автоматический возврат средств. Отмена необратима; возврат денег происходит автоматически на стороне бэкенда как часть успешной отмены, без отдельного клиентского действия.

Условие доступности отмены: до начала класса остаётся более 10 минут (`startsAt − now > 10 мин`, CON-006). Клиентское приложение проверяет порог локально как вспомогательную меру блокировки UI (NFR-013); финальную проверку выполняет бэкенд при обработке запроса.

### User Story

> Как клиент, я хочу отменить свою бронь не позднее чем за 10 минут до начала класса,
> чтобы вернуть деньги, если планы изменились.

### Бизнес-ценность

- Снижение нагрузки на поддержку за счёт самообслуживания при отказе от класса.
- Защита студии от финансовых потерь через жёсткий порог 10 минут (BR-003).
- Автоматизация возврата средств без ручных обращений (BR-010).

## Описание логики

### Шаг 1. Проверка доступности отмены (до показа действия)

Кнопка «Отменить бронь» на SCR-009 доступна только при `Booking.status = active`. Дополнительно применяется локальная проверка порога 10 минут по [LOGIC-002](LOGIC-002-slot-availability.md): если `startsAt − now ≤ 10 мин`, кнопка переведена в недоступное состояние с указанием причины (NFR-013 — мера UX, не единственный механизм защиты). Локальная проверка не блокирует отправку запроса насильно при расхождении часов; финальное решение принимает бэкенд.

### Шаг 2. Модальное подтверждение

Тап по доступной кнопке «Отменить бронь» открывает модальное окно поверх SCR-009. Модальное окно однозначно сообщает о необратимости действия и об автоматическом возврате средств до нажатия подтверждающей кнопки. Закрытие модалки или кнопка «Не отменять» прерывают флоу без отправки запроса.

### Шаг 3. Вызов API и обработка ответа

Подтверждение в модалке инициирует `POST /bookings/{bookingId}/cancel` (`cancelBooking`). На время запроса подтверждающая кнопка переводится в состояние загрузки, повторный тап (double-submit) блокируется. По ответу:

- **200 OK** — возвращена обновлённая `Booking`: `status = cancelled_by_client`, вложенный `Payment.status = refunded` (FR-016). Возврат инициирован бэкендом автоматически (BR-010). Показывается снек успеха, осуществляется возврат на SCR-008.
- **409 `cancellation_not_allowed`** — порог 10 минут пройден на стороне бэкенда. Показывается снек «Отмена брони доступна не позднее чем за 10 минут до начала класса», модалка закрывается, бронь ре-фетчится (переход в состояние «отмена недоступна»).
- **404** — бронь не найдена или больше не принадлежит клиенту. Показывается снек «Бронь не найдена», возврат на SCR-008.
- **401** — переход на SCR-001 по [LOGIC-001](LOGIC-001-session.md).
- **5xx / нет сети** — снек стандартного формата, модалка остаётся открытой, пользователь может повторить попытку.

## API запросы

### POST /bookings/{bookingId}/cancel

**Триггер:** Подтверждение в модальном окне «Отменить бронь» на SCR-009.

**Спецификация:** [openapi.yaml](../../api/openapi.yaml) → `cancelBooking` (POST /bookings/{bookingId}/cancel)

**Headers:**

| Поле | Описание |
|------|----------|
| `Authorization` | `Bearer <JWT>` — токен клиента |

**Параметры/Body:**

| Параметр | Тип | Описание | Значение/Источник |
|----------|-----|----------|-------------------|
| `bookingId` | string (uuid) | Идентификатор брони | Из входных данных SCR-009 |

Тело запроса отсутствует.

**Обработка ответа:**

| Результат | Действие |
|-----------|----------|
| Загрузка | Лоадер на подтверждающей кнопке модалки; блокировка повторного тапа |
| Успех (200) | `Booking.status = cancelled_by_client`, `Payment.status = refunded`; снек «Бронь отменена. Деньги вернутся автоматически.»; возврат на SCR-008 |
| 401 | Переход на SCR-001 (LOGIC-001) |
| 404 | Снек «Бронь не найдена»; возврат на SCR-008 |
| 409 `cancellation_not_allowed` | Снек «Отмена брони доступна не позднее чем за 10 минут до начала класса»; закрыть модалку; ре-фетч брони |
| 5xx | Снек «Произошла ошибка. Попробуйте позже»; остаться в модалке |
| Ошибка сети | Снек «Нет соединения. Проверьте подключение к интернету»; остаться в модалке, оставить retry |

## Локальное хранение

Не требуется. Логика не сохраняет данные локально. Источником истины после отмены является бэкенд; экран SCR-009 актуализирует данные через `getBookingById` при необходимости.

## Критерии приёмки

| ID | Критерий |
|----|----------|
| AC-001 | **Дано** активная бронь (`status = active`), до `startsAt` более 10 минут, **Когда** клиент тапает «Отменить бронь» и подтверждает в модалке, **Тогда** `POST /bookings/{id}/cancel` возвращает 200, `Booking.status` становится `cancelled_by_client`, `Payment.status` — `refunded`, показывается снек об успехе и возврате, осуществляется возврат на SCR-008. |
| AC-002 | **Дано** активная бронь, до `startsAt` менее 10 минут (локально и/или по ответу бэкенда), **Когда** клиент пытается отменить, **Тогда** кнопка подтверждения недоступна с указанием причины; при поступлении `409 cancellation_not_allowed` показывается снек «Отмена брони доступна не позднее чем за 10 минут до начала класса», модалка закрывается, бронь ре-фетчится. |
| AC-003 | **Дано** бронь уже отменена (`status` ≠ `active`), **Когда** экран открыт, **Тогда** кнопка и модалка отмены не показываются; случайная повторная попытка исключена. |
| AC-004 | **Дано** потеря сети во время запроса отмены, **Когда** бэкенд недоступен, **Тогда** показывается снек «Нет соединения…», модалка остаётся открытой, повторная попытка доступна, повторной отправки на ту же попытку не происходит (double-submit заблокирован). |
| AC-005 | **Дано** успешная отмена, **Когда** получен 200, **Тогда** возврат средств инициирован автоматически как часть ответа, отдельного действия «запросить возврат» не требуется (FR-016, BR-010). |
| AC-006 | **Дано** в модалке нажата «Не отменять» или модалка закрыта, **Когда** подтверждение не дано, **Тогда** запрос `cancel` не отправляется, бронь остаётся без изменений. |
| AC-007 | **Дано** отмена возвращает `404`, **Когда** бронь не найдена, **Тогда** показывается снек «Бронь не найдена» и осуществляется возврат на SCR-008. |

## Обработка ошибок

| Тип ошибки | Контекст | Действие |
|------------|----------|----------|
| `cancellation_not_allowed` (409) | Порог 10 минут пройден на бэкенде | Снек с точным текстом из ответа; ре-фетч брони; переход в состояние «отмена недоступна» |
| `404` | Бронь не найдена / не принадлежит клиенту | Снек «Бронь не найдена»; возврат на SCR-008 |
| `401` | Токен недействителен/отсутствует | Переход на SCR-001 (LOGIC-001) |
| `5xx` | Внутренняя ошибка бэкенда | Снек «Произошла ошибка. Попробуйте позже»; модалка открыта, retry доступен |
| Нет сети | Запрос не дошёл | Снек «Нет соединения…»; модалка открыта, retry доступен |
| Double-submit | Повторный тап во время запроса | Блокировка повторной отправки лоадером на кнопке |
