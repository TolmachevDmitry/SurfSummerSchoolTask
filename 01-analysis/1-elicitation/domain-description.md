# Описание предметной области — «Кулинарная студия», клиентское приложение

> Скоуп: только роль **Клиент** — клиентское мобильное приложение и API для него.
> Расписание, программы, шефы, платежи-инфраструктура и админ/шеф-интерфейсы — уже
> существующий бэкенд, потребляются через API как источник истины (см. R-004, R-015,
> R-027, R-028 в брифе). Домен ниже описывает сущности и правила **в объёме, видимом
> клиенту**, а не внутреннюю реализацию бэкенда.

## 1. Ключевые сущности

### Client (Клиент)
Пользователь приложения, который просматривает расписание и записывается на классы.

- id
- имя, телефон/email (в зависимости от способа авторизации)
- признак «постоянный клиент» (если решено включать в MVP — см. открытый вопрос)
- список аллергий/ограничений по умолчанию (если профильное поле — открытый вопрос)

### ClassProgram (Программа / меню класса)
Шаблон кулинарного класса: что готовят, для какого уровня.

- id, название («Простая итальянская кухня», «Продвинутая французская» и т.п.)
- уровень сложности (новичок / опытный)
- признак «требует духовки/сложной техники» — влияет на ограничение вместимости
  (не больше 8 человек, а не 12, как для обычных столов)
- состав меню (что готовится)
- длительность (у Артёма — как правило ~3 часа)

*Программы приходят из API как справочник, клиентское приложение их не создаёт и не
редактирует.*

### Chef (Шеф)
Ведущий класса.

- id, имя, фото, рейтинг (агрегированная оценка клиентов)
- статус (штатный / приглашённый — не обязательно видно клиенту, зависит от API)

### ClassSlot (Слот / сеанс класса)
Конкретное проведение программы в конкретное время конкретным шефом — единица,
на которую бронируется клиент.

- id
- дата и время начала, длительность
- программа (ссылка на ClassProgram)
- шеф (ссылка на Chef)
- вместимость (max_seats): 12 по умолчанию, либо 8, если программа требует духовки/
  сложной техники
- количество свободных мест (available_seats) — вычисляется бэкендом, приложение
  только отображает и обрабатывает отказ при бронировании, если мест не осталось
  (гарантия «0 двойных броней» — на стороне бэкенда, вне скоупа клиента)
- статус слота: `scheduled` (открыт для записи) / `cancelled_by_studio` (отменён
  студией — форс-мажор или решение организатора, см. R-008) / возможно `completed`,
  `in_progress` — если нужно клиенту для истории (уточнить)
- горизонт отображения клиенту: по умолчанию ближайшие 7 дней, с возможностью
  расширить фильтром дат (R-027); при отсутствии слотов — empty state «Пока нет
  доступных классов»

### RentalKit (Прокатный набор экипировки)
Фартук + набор ножей, который студия может выдать напрокат.

- тариф проката (приходит из API — R-015)
- свободный прокатный фонд на конкретный слот (сколько наборов ещё доступно)

### Booking (Бронь)
Связь клиента и слота — центральная сущность клиентского сценария.

- id
- клиент (ссылка на Client)
- слот (ссылка на ClassSlot)
- выбор экипировки: «своя» / «прокатная» (если прокатная — привязка к RentalKit,
  ограничена свободным прокатным фондом слота)
- аллергии/ограничения, указанные для этой брони (если поле привязано к брони, а не
  только к профилю)
- статус брони:
  - `active` / `confirmed` — подтверждена
  - `cancelled_by_client` — отменена клиентом (в допустимый срок)
  - `cancelled_by_studio` — отменена студией, с указанием причины; **повторная запись
    на этот слот запрещена**, клиенту приходит push-уведомление (R-008)
  - возможно `no_show` / `late_cancellation` — если вводится штрафной статус
    (открытый вопрос)
  - `completed` — класс прошёл, можно оставить оценку
- время создания брони, время последней отмены (для проверки допустимости отмены
  по временному порогу)

### Rating (Оценка шефа)
Оценка, которую клиент оставляет после посещённого класса.

- id
- бронь, к которой привязана оценка (обязательная связь — оценить можно только то,
  что реально посетил)
- клиент, шеф (через бронь)
- числовая оценка (звёзды), опционально текстовый отзыв
- дата оценки

### Notification (Уведомление)
Сообщение клиенту о событиях, связанных с его бронями.

- тип: напоминание о предстоящем классе, подтверждение брони, отмена студией
  (обязательный тип по R-008), возможно — приглашение оценить шефа после класса
- канал: push (минимально гарантирован для отмены студией); email/SMS — под вопросом
- связь с конкретной бронью/слотом

## 2. Отношения между сущностями

```
Client 1 ── * Booking * ── 1 ClassSlot * ── 1 ClassProgram
                │                    * ── 1 Chef
                │
                ├── 0..1 RentalKit (через выбор экипировки)
                └── 0..1 Rating (после завершения)

ClassSlot ── 0..1 причина отмены (если cancelled_by_studio)
Notification * ── 1 Client
Notification * ── 0..1 Booking / ClassSlot (контекст события)
```

## 3. Жизненный цикл брони (Booking)

```
                  запись клиента
                  (если available_seats > 0)
[нет брони] ────────────────────────────▶ active/confirmed
                                              │
              отмена клиентом                │            класс состоялся
              (в допустимый срок) ◀──────────┼──────────▶ completed
                     │                        │                 │
                     ▼                        │                 ▼
            cancelled_by_client                │             можно оставить Rating
                                               │
                            студия отменяет слот (форс-мажор
                            / решение организатора)
                                               │
                                               ▼
                                   cancelled_by_studio
                                   (push клиенту, причина указана,
                                    повторная запись на слот запрещена)
```

Открытый момент (см. вопросы к заказчику): нужен ли отдельный терминальный статус
для «клиент отменил слишком поздно» / «клиент не пришёл» — сейчас в брифе это описано
как проблема бизнеса («продукты закуплены и пропадают»), но не как явное состояние
брони.

## 4. Ключевые бизнес-правила, релевантные клиентскому приложению

1. **Вместимость слота зависит от программы.** Обычный слот — до 12 человек, слот с
   духовкой/сложной техникой — до 8. Приложение не вычисляет это само, а отображает
   `max_seats`/`available_seats`, отданные бэкендом.
2. **Гарантия от двойного бронирования — вне клиента.** Приложение отправляет запрос
   на бронирование и корректно обрабатывает отказ, если мест не осталось (гонка на
   стороне бэкенда решена атомарно, R-004).
3. **Ограничение поздней отмены.** Должен существовать временной порог, после которого
   клиент не может бесплатно отменить бронь (в брифе ориентир — «за 10 минут уже
   проблема»); точное значение — открытый вопрос к заказчику.
4. **Отмена по инициативе студии не удаляет бронь, а меняет её статус.** Клиент видит
   бронь со статусом «Отменена студией» и причиной, получает push, повторно записаться
   на этот же слот не может (R-008).
5. **Прокат экипировки ограничен фондом слота.** Число доступных прокатных наборов
   можно исчерпать независимо от общей вместимости слота по местам.
6. **Оценка привязана к факту посещения.** Оставить Rating можно только по завершённой
   брони (`completed`), не «просто шефу».
7. **Горизонт видимости расписания — 7 дней по умолчанию**, расширяемый фильтром дат;
   пустой список должен сопровождаться понятным empty state, а не просто пустым
   экраном (R-027).
8. **Клиент не создаёт и не редактирует справочные данные.** Программы, шефы, слоты,
   тарифы проката — исключительно read-only для клиента; источник — API бэкенда.

## 5. Явно вне скоупа текущей поставки

- Онлайн-оплата (пока нал/перевод на месте, если не решено иначе).
- Интерфейсы шефа и администратора/владельца — уже существуют отдельно.
- Создание/редактирование расписания, программ, состава шефов.
- Внутренняя логика формирования расписания и транзакционность бронирования на
  бэкенде (SLA, атомарность) — рассматривается как black-box.
- Миграция исторических данных (проект без легаси-данных, R-015).

## 6. Открытые вопросы, влияющие на модель домена

Полный список — в файле `2-clarifying-questions.md`. Наиболее значимые для домена:

- Нужен ли статус брони «поздняя отмена / no-show» как отдельная сущность/атрибут.
- Аллергии — поле профиля клиента или поле конкретной брони (или оба).
- «Постоянный клиент» — существующий атрибут из API или фича для проектирования.
- Нужен ли waitlist при заполненном слоте.
- Публичность текстовых отзывов в Rating.
