# Описание предметной области — «Кулинарная студия», клиентское приложение

> Скоуп: только роль **Клиент** — клиентское мобильное приложение (кроссплатформенное,
> React Native/Flutter) и API для него. Расписание, программы, шефы, платежная
> инфраструктура (эквайринг) и админ/шеф-интерфейсы — уже существующий бэкенд,
> потребляются через API как источник истины (см. R-004, R-015, R-027, R-028 в брифе).
> Домен ниже описывает сущности и правила **в объёме, видимом клиенту**, а не
> внутреннюю реализацию бэкенда.

## 1. Суть домена

Кулинарная студия проводит очные мастер-классы: шеф в определённое время ведёт группу
клиентов по конкретной программе (что готовят, какого уровня сложности). Клиент через
мобильное приложение находит подходящий по времени и теме класс, бронирует место,
при необходимости берёт напрокат фартук и нож, оплачивает участие онлайн и приходит
в студию. После класса он может оставить публичную оценку шефу. Ключевое ограничение
бизнеса — вместимость слота (обычно до 12 человек, до 8 — если нужна духовка/сложная
техника) и жёсткий порог отмены за 10 минут до начала, чтобы студия не закупала
продукты впустую. Клиентское приложение не управляет расписанием и не является
источником истины ни по одной справочной сущности (программы, шефы, слоты, тарифы) —
оно только отображает данные бэкенда и отправляет клиентские действия (бронь, отмена,
оплата, оценка).

## 2. Действующие лица

- **Клиент** — единственная роль, для которой проектируется приложение в этой
  поставке. Просматривает расписание (после логина), бронирует слот, выбирает
  экипировку, оплачивает участие онлайн, получает push-уведомления, оставляет
  публичную оценку и отзыв шефу, видит историю прошлых классов.
- **Шеф** — ведёт класс очно. В клиентском приложении присутствует только как
  справочная сущность (карточка, рейтинг, публичные отзывы) — сам шеф приложением
  не пользуется, его интерфейс существует отдельно и в скоуп не входит.
- **Студия / администратор** — управляет расписанием, программами, составом шефов
  и прокатным фондом, может отменить слот (`cancelled_by_studio`) через собственный
  админ-интерфейс. В клиентском приложении не участвует напрямую, но его действия
  (отмена слота, изменение расписания) отражаются на клиенте через API и push-
  уведомления.
- **Бэкенд / API** — системный актор, а не пользователь: источник истины по всем
  справочным сущностям (программы, шефы, слоты, тарифы проката), гарант атомарности
  бронирования и обработчик онлайн-оплаты через существующую платёжную
  инфраструктуру. Клиентское приложение только потребляет API и не дублирует его
  логику.

## 3. Ключевые сущности

### Client (Клиент)
Пользователь приложения, который просматривает расписание и записывается на классы.
Входит в приложение по логину и паролю, имеет имя и контактные данные. В профиле
хранится список аллергий по умолчанию — чек-лист типовых аллергенов, который
подтягивается автоматически при каждой новой брони (см. Booking).

Признак «постоянный клиент» в текущую поставку не включается — фича отложена на
следующую итерацию, поле в профиле не проектируем.

Доступ к расписанию только для авторизованных пользователей. Гостевого просмотра нет:
логин обязателен ещё до просмотра расписания, а не только на шаге бронирования. Это
первый экран приложения.

### ClassProgram (Программа / меню класса)
Шаблон кулинарного класса: что готовят, для какого уровня. Помимо названия
(«Простая итальянская кухня», «Продвинутая французская» и т.п.), уровня сложности
(новичок / опытный) и состава меню, у программы есть признак «требует духовки/сложной
техники» — он влияет на ограничение вместимости слота (не больше 8 человек, а не 12,
как для обычных столов), длительность (как правило ~3 часа) и стоимость участия,
которая отображается клиенту на экранах слота и бронирования.

*Программы приходят из API как справочник, клиентское приложение их не создаёт и не
редактирует.*

### Chef (Шеф)
Ведущий класса. В клиентском приложении представлен карточкой с именем, фото и
рейтингом — агрегированной оценкой клиентов. Статус (штатный / приглашённый) в
модели может присутствовать, но не обязательно виден клиенту — зависит от того, что
отдаёт API.

### ClassSlot (Слот / сеанс класса)
Конкретное проведение программы в конкретное время конкретным шефом — единица,
на которую бронируется клиент. У слота есть дата и время начала, длительность,
ссылка на программу (с её стоимостью) и на шефа. Вместимость (max_seats) — 12 по
умолчанию, либо 8, если программа требует духовки/сложной техники. Количество
свободных мест (available_seats) вычисляется бэкендом — приложение только отображает
его и обрабатывает отказ при бронировании, если мест не осталось (гарантия «0 двойных
броней» — на стороне бэкенда, вне скоупа клиента).

Слот может находиться в одном из состояний: открыт для записи (`scheduled`),
отменён студией (`cancelled_by_studio` — форс-мажор или решение организатора, см.
R-008), идёт прямо сейчас (`in_progress`) или уже прошёл (`completed` — нужно для
истории и открытия оценки).

Клиенту слоты показываются на горизонте ближайших 7 дней по умолчанию, с возможностью
расширить его фильтром дат (R-027); при отсутствии слотов — empty state «Пока нет
доступных классов».

Лист ожидания не нужен. Если мест нет, слот просто отображается как недоступный для
записи (available_seats = 0), без очереди/waitlist.

### RentalKit (Прокатный набор экипировки)
Фартук + набор ножей, который студия может выдать напрокат. Тариф проката приходит
из API (R-015) и отображается клиенту; там же виден свободный прокатный фонд на
конкретный слот — сколько наборов ещё доступно.

Выбор экипировки — обязательный шаг бронирования. Клиент обязан явно выбрать «своя
экипировка» или «прокатная» при оформлении брони — пропустить этот шаг и решить на
месте нельзя. Если прокатный фонд слота исчерпан, вариант «прокатная» полностью
блокируется в UI (а не просто помечается предупреждением).

### Booking (Бронь)
Связь клиента и слота — центральная сущность клиентского сценария. Обязательно
фиксирует выбор экипировки («своя» / «прокатная»; если прокатная — привязку к
RentalKit в пределах свободного прокатного фонда слота) и статус оплаты (оплата —
обязательная предпосылка создания самой брони, см. раздел 3a «Оплата»). Аллергии
для брони отдельно не хранятся — они подтягиваются автоматически из профиля клиента
(см. Client), так как решено хранить их именно в профиле, а не на уровне каждой брони.

Возможные состояния брони и переходы между ними (подтверждена сразу без
промежуточного статуса «заявка отправлена», отмена клиентом, отмена студией,
завершение) подробно описаны в разделе 5 «Жизненный цикл брони» ниже. Отдельного
статуса «поздняя отмена» / «no-show» не вводится: после порога в 10 минут клиент
технически не может инициировать отмену, и бронь остаётся активной.

Порог отмены — ровно 10 минут до начала класса. Это жёсткая граница: менее чем за
10 минут до старта клиент не может ни записаться на слот, ни отменить существующую
бронь через приложение. UI должен скрывать/дизейблить обе кнопки после наступления
порога.

Групповая бронь не нужна. Один аккаунт клиента = одно место в одной брони.
Бронирование класса на нескольких человек с одного аккаунта не проектируем.

### Payment (Оплата)

- Оплата — **синхронная и обязательная предпосылка создания брони**, а не отдельный
  шаг после неё. Технически это токенизация карты в приложении с мгновенным ответом
  (успех/ошибка), а не редирект на страницу провайдера с последующим вебхуком.
  Асинхронного «периода ожидания» между действием клиента и результатом оплаты нет.
- Если оплата не прошла (ошибка списания) — **Booking не создаётся вообще**, место
  на слоте не резервируется даже временно. Клиент видит ошибку оплаты на экране
  оформления брони и может повторить попытку (новая попытка = новая независимая
  попытка оплаты, а не изменение статуса уже существующей брони).
- Из-за этого промежуточный статус `pending` доменной модели не нужен: раз Booking
  физически не может существовать без успешной оплаты, состояния «бронь создана, но
  оплата ещё не прошла» в модели не возникает.
- Статус оплаты брони — только два значения:
  - `paid` — оплата прошла успешно, Booking создан вместе с ней (создаются атомарно,
    одним действием клиента).
  - `refunded` — деньги возвращены клиенту после отмены оплаченной брони.
- Возврат средств (`refunded`) происходит **автоматически** в двух случаях:
  - отмена брони самим клиентом (`cancelled_by_client`, в пределах порога 10 минут);
  - отмена слота студией (`cancelled_by_studio`) — если бронь на этот слот была
    оплачена, деньги возвращаются клиенту автоматически, без обращения в поддержку.
  Ручного/через-поддержку сценария возврата в этой поставке нет.
- Провайдер эквайринга и точная реализация токенизации/возврата остаются на стороне
  существующей бэкенд-платёжной инфраструктуры (вне скоупа клиентского приложения) —
  но сам факт синхронности флоу и набор статусов (`paid`/`refunded`) для доменной
  модели уже зафиксирован и больше не является открытым вопросом.
- Клиентское приложение показывает стоимость (класс + прокат, если выбран) до оплаты,
  на экране оформления брони, до того как клиент подтверждает и оплачивает.

### Rating (Оценка шефа)
Оценка, которую клиент оставляет после посещённого класса — числовая (звёзды) и
текстовый отзыв. Обязательно привязана к конкретной завершённой брони: оценить можно
только то, что клиент реально посетил, а связь с шефом определяется через эту бронь.

Текстовые отзывы публичны — видны другим клиентам приложения (например, в карточке
шефа), не только внутренние данные для студии. Оценка становится доступна сразу
после окончания класса по времени (по факту наступления конца слота / перехода
ClassSlot в `completed`), без дополнительного push-триггера или ручного открытия из
истории. Оценку нельзя редактировать задним числом — после отправки она фиксируется
как финальная.

### Notification (Уведомление)
Сообщение клиенту о событиях, связанных с его бронями — всегда привязано к конкретной
брони/слоту. Обязательные типы на старте: напоминание о предстоящем классе (за
2–3 часа до начала), подтверждение брони и отмена студией (обязательна по R-008).
Канал — только push в MVP, email/SMS в текущую поставку не входят.

Настройки уведомлений клиентом не предусмотрены. Логика уведомлений фиксированная
(нет toggle вкл/выкл, нет выбора времени напоминания пользователем).

## 3a. Оплата — сводка правил

1. Онлайн-оплата входит в текущую поставку — это отменяет прежнее допущение «оплата
   остаётся оффлайн».
2. Стоимость класса и стоимость проката экипировки отображаются клиенту ещё до
   оплаты, на экране слота/оформления брони.
3. Оплата происходит **синхронно и сразу при бронировании** — Booking и успешная
   оплата создаются одним атомарным действием клиента, отдельного шага «оплатить
   позже» нет.
4. Если оплата не проходит — **Booking не создаётся**, место не резервируется;
   клиент может повторить попытку оплаты.
5. Статусы Payment — только `paid` и `refunded`; промежуточный `pending` и отдельный
   персистентный `failed` доменной модели не нужны (см. раздел «Payment» выше).
6. Возврат средств (`refunded`) — всегда автоматический: при отмене брони клиентом
   (в пределах порога 10 минут) и при отмене оплаченного слота студией
   (`cancelled_by_studio`).

## 4. Отношения между сущностями

```
Client 1 ── * Booking * ── 1 ClassSlot * ── 1 ClassProgram
                │                    * ── 1 Chef
                │
                ├── 0..1 RentalKit (через выбор экипировки, обязательный шаг)
                ├── 1 Payment (обязательна: Booking не существует без успешной
                │              оплаты — создаются атомарно; статус paid → refunded)
                └── 0..1 Rating (после завершения, публичный отзыв)

Client 1 ── 1 профиль аллергий (чек-лист, подтягивается в каждую Booking)

ClassSlot ── 0..1 причина отмены (если cancelled_by_studio)
Notification * ── 1 Client
Notification * ── 0..1 Booking / ClassSlot (контекст события)
```

## 5. Жизненный цикл брони (Booking)

```
                  запись клиента + оплата
                  (если available_seats > 0,
                   выбор экипировки обязателен,
                   оплата проходит синхронно —
                   при её неуспехе Booking НЕ
                   создаётся, попытку можно повторить)
[нет брони] ────────────────────────────▶ active/confirmed ── (сразу, без
                                              │  Payment: paid    промежуточного
                                              │                   статуса)
              отмена клиентом                │            класс состоялся
              (не позднее чем за 10 минут    │            (ClassSlot → completed)
               до начала) ◀───────────────────┼──────────▶
                     │                        │                 │
                     ▼                        │                 ▼
            cancelled_by_client                │       доступна оценка (Rating)
            Payment: paid → refunded            │       сразу по времени окончания
            (возврат автоматический)
                            студия отменяет слот (форс-мажор
                            / решение организатора)
                                               │
                                               ▼
                                   cancelled_by_studio
                                   Payment: paid → refunded
                                   (возврат автоматический)
                                   (push клиенту, причина указана,
                                    повторная запись на слот запрещена)
```

После наступления порога в 10 минут до начала класса кнопка отмены в приложении
недоступна — бронь остаётся `active`/`confirmed`, отдельного статуса «поздняя
отмена» или «no-show» не вводится в этой поставке.

## 6. Ключевые бизнес-правила, релевантные клиентскому приложению

1. **Вместимость слота зависит от программы.** Обычный слот — до 12 человек, слот с
   духовкой/сложной техникой — до 8. Приложение не вычисляет это само, а отображает
   `max_seats`/`available_seats`, отданные бэкендом.
2. **Гарантия от двойного бронирования — вне клиента.** Приложение отправляет запрос
   на бронирование и корректно обрабатывает отказ, если мест не осталось (гонка на
   стороне бэкенда решена атомарно, R-004).
3. **Жёсткий порог отмены — 10 минут до начала класса.** После этого момента ни
   запись, ни отмена через приложение недоступны. Отдельного штрафного статуса за
   позднюю отмену/неявку нет.
4. **Отмена по инициативе студии не удаляет бронь, а меняет её статус.** Клиент видит
   бронь со статусом «Отменена студией» и причиной, получает push, повторно записаться
   на этот же слот не может (R-008).
5. **Прокат экипировки ограничен фондом слота и обязателен как шаг выбора.** Выбор
   «своя/прокатная» экипировка — обязательная часть оформления брони; если прокатный
   фонд слота исчерпан, вариант проката блокируется полностью (не просто
   предупреждение).
6. **Оценка привязана к факту посещения и доступна сразу по завершении.** Оставить
   Rating (звёзды + текстовый отзыв, публичный) можно только по завершённой брони
   (`completed`), сразу как класс закончился по времени; редактирование оценки после
   отправки не допускается.
7. **Горизонт видимости расписания — 7 дней по умолчанию**, расширяемый фильтром дат;
   пустой список должен сопровождаться понятным empty state, а не просто пустым
   экраном (R-027).
8. **Клиент не создаёт и не редактирует справочные данные.** Программы, шефы, слоты,
   тарифы проката — исключительно read-only для клиента; источник — API бэкенда.
9. **Доступ к приложению только после логина.** Просмотр расписания без авторизации
   недоступен; способ входа — логин и пароль.
10. **Онлайн-оплата — обязательная и синхронная предпосылка создания брони**
    (см. раздел 3a): бронь и успешная оплата возникают одним атомарным действием,
    при неуспешной оплате Booking не создаётся, а не переходит в промежуточный статус.
11. **Групповых броней нет** — один аккаунт бронирует ровно одно место.
12. **Листа ожидания нет** — заполненный слот просто отображается как недоступный.
13. **Клиенту доступна история прошлых (завершённых) классов**, а не только активные
    будущие брони — отдельный раздел «История» в приложении.
14. **Возврат средств — всегда автоматический.** При отмене оплаченной брони клиентом
    (в пределах порога 10 минут) или при отмене оплаченного слота студией статус
    Payment переходит `paid → refunded` без ручного вмешательства поддержки.

## 7. Явно вне скоупа текущей поставки

- Интерфейсы шефа и администратора/владельца — уже существуют отдельно.
- Создание/редактирование расписания, программ, состава шефов.
- Внутренняя логика формирования расписания и транзакционность бронирования на
  бэкенде (SLA, атомарность) — рассматривается как black-box.
- Миграция исторических данных (проект без легаси-данных, R-015).
- Мультиязычность/мультивалютность — интерфейс одноязычный (русский).
- Возрастные ограничения / согласие родителя на запись — не требуются, все классы
  считаются 18+.
- Лист ожидания (waitlist) при заполненном слоте.
- Групповая бронь на нескольких человек с одного аккаунта.
- Отдельный статус брони «поздняя отмена» / «no-show».
- Фича «постоянный клиент» — отложена на следующую итерацию.
- Настройка уведомлений клиентом (вкл/выкл, время).
