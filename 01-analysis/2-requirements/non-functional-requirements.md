# Нефункциональные требования (НФТ)

> Описывают качественные атрибуты и системные ограничения («как система это делает»),
> в отличие от функциональных требований («что система делает»), вынесенных в
> `functional-requirements.md`.

| ID | Формулировка | Приоритет | Источник |
|---|---|---|---|
| NFR-001 | Клиентское приложение должно быть кроссплатформенным (React Native/Flutter), с единой кодовой базой для мобильных платформ. | Must | Эпиграф домена «Скоуп» |
| NFR-002 | Интерфейс приложения — одноязычный (русский); мультивалютность и мультиязычность не поддерживаются. | Must | §7 «Явно вне скоупа» |
| NFR-003 | Клиентское приложение архитектурно не является источником истины ни для одной справочной сущности (программы, шефы, слоты, тарифы) — оно строго read-only консьюмер API бэкенда. | Must | §1; §6, п.8 |
| NFR-004 | Гарантия атомарности бронирования и защита от двойного бронирования обеспечиваются на стороне бэкенда; клиент лишь корректно обрабатывает отказ, не реализуя собственную блокирующую логику. | Must | §6, п.2; R-004 |
| NFR-005 | Оплата должна происходить синхронно, с мгновенным откликом (успех/ошибка) в рамках одного пользовательского действия, без периода асинхронного ожидания (без редиректа на страницу провайдера и последующего вебхука). | Must | §3a, п.1 |
| NFR-006 | Логика уведомлений фиксирована и не подлежит пользовательской настройке (нет toggle вкл/выкл, нет выбора времени напоминания). | Must | §3 «Notification» |
| NFR-007 | В MVP поддерживается только один канал доставки уведомлений — push; email/SMS не реализуются. | Must | §3 «Notification» |
| NFR-008 | Токенизация карты, работа с провайдером эквайринга и обработка возврата средств реализуются существующей бэкенд-платёжной инфраструктурой; клиент не хранит платёжные реквизиты и не содержит собственной логики эквайринга. | Must | §3a (сводка) |
| NFR-009 | Возрастная верификация / согласие родителя не реализуются — все классы рассматриваются как 18+ по умолчанию. | Must | §7 «Явно вне скоупа» |
| NFR-010 | Проект не предусматривает миграцию исторических (легаси) данных. | Must | §7; R-015 |

## Примечания
- Границы между ФТ и НФТ проведены по принципу: если требование описывает конкретное
  действие/реакцию системы на ввод пользователя — это ФТ; если оно описывает свойство
  системы в целом (архитектурное ограничение, канал, локализация, синхронность
  протокола) — это НФТ.
- Требования к производительности (SLA, время отклика в мс), безопасности хранения
  данных, отказоустойчивости и нагрузке в исходном описании домена не заданы —
  такие пункты не выведены, чтобы не добавлять того, чего нет в источнике.
