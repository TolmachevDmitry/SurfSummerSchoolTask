# Ограничения и скоуп

## Часть 1. Ограничения (технические и бизнес-ограничения)

| ID | Формулировка | Приоритет | Источник |
|---|---|---|---|
| CON-001 | Клиентское приложение не является источником истины ни для одной справочной сущности (программы, шефы, слоты, тарифы проката) — оно только отображает данные бэкенда и отправляет клиентские действия. | Must | §1 «Суть домена» |
| CON-002 | Гарантия атомарности бронирования и защита «0 двойных броней» реализуются на стороне бэкенда; клиентское приложение не реализует собственную блокирующую логику, а лишь обрабатывает отказ. | Must | §6, п.2; R-004 |
| CON-003 | Провайдер эквайринга, токенизация карты и техническая реализация возврата средств остаются в существующей бэкенд-платёжной инфраструктуре — вне скоупа клиентского приложения. | Must | §3a (сводка) |
| CON-004 | Расписание, программы, состав шефов и прокатный фонд управляются исключительно через существующий административный интерфейс, отдельный от клиентского приложения. | Must | §2 «Студия / администратор» |
| CON-005 | Вместимость слота (max_seats) жёстко определяется признаком программы «требует духовки/сложной техники»: 12 человек в обычном случае, 8 — если признак установлен; значение приходит от бэкенда, а не вычисляется клиентом. | Must | §3 «ClassProgram», «ClassSlot»; §6, п.1 |
| CON-006 | Порог отмены записи/брони — ровно 10 минут до начала класса; порог не настраивается и одинаков для всех слотов. | Must | §3 «Booking»; §6, п.3 |
| CON-007 | Модель оплаты содержит только два статуса — `paid` и `refunded`; промежуточные статусы (`pending`, ручной `failed`) в доменной модели отсутствуют по замыслу. | Must | §3a, п.3, п.5 |
| CON-008 | Модель брони не содержит отдельного статуса «поздняя отмена» / «no-show»: после порога в 10 минут бронь остаётся `active`/`confirmed`. | Must | §3 «Booking»; §5 |
| CON-009 | Интерфейс приложения — одноязычный (русский), без поддержки мультивалютности. | Must | §7 |
| CON-010 | Проект реализуется без миграции исторических (легаси) данных. | Must | §7; R-015 |
| CON-011 | Возрастные ограничения и согласие родителя не применяются — все классы считаются 18+. | Must | §7 |

## Часть 2. Скоуп

### 2.1. Что входит в скоуп текущей поставки

| ID | Формулировка | Приоритет | Источник |
|---|---|---|---|
| SCOPE-IN-001 | Клиентское мобильное приложение (кроссплатформенное, React Native/Flutter) и API для роли «Клиент». | Must | Эпиграф домена «Скоуп» |
| SCOPE-IN-002 | Просмотр расписания слотов (после логина), бронирование, выбор экипировки, онлайн-оплата, push-уведомления, публичная оценка/отзыв шефу, история прошлых классов. | Must | §2 «Клиент» |
| SCOPE-IN-003 | Отображение справочных сущностей бэкенда (программы, шефы, слоты, тарифы проката) в режиме read-only. | Must | §1; §6, п.8 |

### 2.2. Что явно вне скоупа текущей поставки

| ID | Формулировка | Приоритет | Источник |
|---|---|---|---|
| SCOPE-OUT-001 | Интерфейсы шефа и администратора/владельца студии — уже существуют отдельно. | — (вне скоупа) | §7 |
| SCOPE-OUT-002 | Создание и редактирование расписания, программ, состава шефов. | — (вне скоупа) | §7 |
| SCOPE-OUT-003 | Внутренняя логика формирования расписания и транзакционность бронирования на бэкенде (SLA, атомарность) — рассматривается как чёрный ящик. | — (вне скоупа) | §7 |
| SCOPE-OUT-004 | Миграция исторических (легаси) данных. | — (вне скоупа) | §7; R-015 |
| SCOPE-OUT-005 | Мультиязычность и мультивалютность интерфейса. | — (вне скоупа) | §7 |
| SCOPE-OUT-006 | Возрастные ограничения / согласие родителя на запись. | — (вне скоупа) | §7 |
| SCOPE-OUT-007 | Лист ожидания (waitlist) при заполненном слоте. | — (вне скоупа) | §7; §3 «ClassSlot» |
| SCOPE-OUT-008 | Групповая бронь на нескольких человек с одного аккаунта. | — (вне скоупа) | §7; §3 «Booking» |
| SCOPE-OUT-009 | Отдельный статус брони «поздняя отмена» / «no-show». | — (вне скоупа) | §7; §3 «Booking» |
| SCOPE-OUT-010 | Фича «постоянный клиент» — отложена на следующую итерацию. | — (вне скоупа) | §7; §3 «Client» |
| SCOPE-OUT-011 | Пользовательская настройка уведомлений (вкл/выкл, выбор времени напоминания). | — (вне скоупа) | §7; §3 «Notification» |

## Примечание
Открытых вопросов, влияющих на доменную модель, на момент составления описания не осталось (§8 источника).
