# Use Case (варианты использования)

> Приоритет — на уровне варианта использования в целом (MoSCoW).
> Актёры: **Клиент** (основной, инициирует сценарии), **Бэкенд/API** (системный
> актор — источник истины и обработчик оплаты/атомарности), **Студия** — инициирует
> отмену слота через отдельный интерфейс, не входящий в скоуп, но её действие
> отражается в сценарии UC-005.

---

## UC-001. Авторизация клиента
**Приоритет:** Must
**Источник:** §3 «Client»; §6, п.9

- **Актёр:** Клиент
- **Предусловие:** Клиент установил приложение, ещё не авторизован.
- **Основной поток:**
  1. Клиент открывает приложение и видит экран логина (первый экран).
  2. Клиент вводит логин и пароль.
  3. Система проверяет учётные данные через бэкенд.
  4. Система открывает доступ к расписанию.
- **Альтернативный поток:** Неверные учётные данные → система показывает ошибку, доступ к расписанию не предоставляется.
- **Постусловие:** Клиент авторизован и видит расписание слотов.

## UC-002. Просмотр расписания классов
**Приоритет:** Must
**Источник:** §3 «ClassSlot»; §6, п.7; R-027

- **Актёр:** Клиент
- **Предусловие:** Клиент авторизован.
- **Основной поток:**
  1. Клиент открывает экран расписания.
  2. Система запрашивает у бэкенда слоты на горизонте 7 дней по умолчанию.
  3. Система отображает карточки слотов (время, программа, шеф, стоимость, available_seats/max_seats, статус).
  4. Клиент опционально расширяет горизонт фильтром по датам — система повторяет запрос с новым диапазоном.
- **Альтернативный поток:** Нет доступных слотов на выбранном горизонте → система показывает empty state «Пока нет доступных классов».
- **Постусловие:** Клиент видит актуальный список слотов, отданный бэкендом.

## UC-003. Бронирование слота с оплатой и выбором экипировки
**Приоритет:** Must
**Источник:** §3 «Booking», «RentalKit», «Payment»; §3a; §5; §6, п.1, 2, 5, 10, 11

- **Актёр:** Клиент; Бэкенд/API (проверка мест, атомарность, оплата)
- **Предусловие:** Клиент авторизован; выбранный слот в статусе `scheduled`; до начала класса более 10 минут; available_seats > 0.
- **Основной поток:**
  1. Клиент открывает карточку слота и нажимает «Записаться».
  2. Система показывает стоимость класса и (если применимо) стоимость проката экипировки.
  3. Клиент обязательно выбирает экипировку: «своя» или «прокатная» (шаг нельзя пропустить).
  4. Клиент подтверждает и вводит данные карты; система выполняет синхронную оплату.
  5. При успешной оплате бэкенд атомарно создаёт Booking со статусом `active/confirmed` и Payment со статусом `paid`.
  6. Система показывает подтверждение брони; клиенту приходит push-подтверждение.
- **Альтернативные потоки:**
  - 3a. Прокатный фонд слота исчерпан → вариант «прокатная» заблокирован в UI, доступен только выбор «своя».
  - 4a. Мест не осталось на момент подтверждения (гонка на бэкенде) → система показывает отказ, Booking не создаётся.
  - 4b. Оплата не прошла → Booking не создаётся, место не резервируется даже временно; система показывает ошибку оплаты; клиент может повторить попытку (новая независимая попытка).
- **Постусловие (успех):** Booking `active/confirmed`, Payment `paid`, аллергии клиента автоматически подтянуты из профиля в бронь, отправлен push-подтверждение.
- **Ограничение:** один аккаунт клиента = одно место в одной брони (без групповой брони).

## UC-004. Отмена брони клиентом
**Приоритет:** Must
**Источник:** §3 «Booking»; §3a, п.6; §5; §6, п.3, 14

- **Актёр:** Клиент
- **Предусловие:** У клиента есть Booking в статусе `active/confirmed`; до начала класса осталось более 10 минут.
- **Основной поток:**
  1. Клиент открывает свою бронь и нажимает «Отменить».
  2. Система переводит Booking в статус `cancelled_by_client`.
  3. Система автоматически инициирует возврат оплаты: Payment `paid → refunded`.
  4. Место слота освобождается (available_seats пересчитывается бэкендом).
- **Альтернативный поток:** До начала класса осталось менее 10 минут → кнопка отмены недоступна (скрыта/задизейблена), отмена невозможна, Booking остаётся `active/confirmed`.
- **Постусловие:** Booking `cancelled_by_client`, Payment `refunded`.

## UC-005. Отображение отмены слота студией
**Приоритет:** Must
**Источник:** §3 «ClassSlot»; §3a, п.6; §6, п.4, 14; R-008

- **Актёр:** Студия (инициирует через отдельный, не входящий в скоуп интерфейс); Клиент (наблюдатель); Бэкенд/API
- **Предусловие:** Слот в статусе `scheduled`, у клиента есть оплаченный Booking на этот слот; студия инициировала отмену слота в своём интерфейсе.
- **Основной поток:**
  1. Бэкенд переводит слот в статус `cancelled_by_studio` с указанием причины.
  2. Бэкенд автоматически переводит связанные оплаченные Booking-и: Payment `paid → refunded`.
  3. Система отправляет клиенту push-уведомление об отмене.
  4. Клиент открывает бронь и видит статус «Отменена студией» с причиной.
- **Постусловие:** Booking `cancelled_by_studio`, Payment `refunded`; повторная запись клиента на этот слот заблокирована.

## UC-006. Оставление оценки и отзыва по завершённому классу
**Приоритет:** Should
**Источник:** §3 «Rating»; §6, п.6

- **Актёр:** Клиент
- **Предусловие:** Booking клиента в статусе, связанном со слотом `completed` (класс завершился по времени); оценка ещё не оставлена.
- **Основной поток:**
  1. Система делает доступной возможность оценки сразу по достижении времени окончания слота.
  2. Клиент открывает завершённую бронь и выставляет оценку (звёзды) и текстовый отзыв.
  3. Система сохраняет Rating, привязанный к Booking (и через неё — к шефу).
  4. Текстовый отзыв становится публично видимым другим клиентам (например, в карточке шефа).
- **Альтернативный поток:** Попытка отредактировать уже отправленную оценку → система запрещает редактирование, оценка зафиксирована как финальная.
- **Постусловие:** Rating создан и опубликован; шеф видит агрегированный рейтинг обновлённым.

## UC-007. Просмотр истории прошлых классов
**Приоритет:** Could
**Источник:** §6, п.13

- **Актёр:** Клиент
- **Предусловие:** Клиент авторизован, имеет хотя бы одну бронь со слотом в статусе `completed`.
- **Основной поток:**
  1. Клиент открывает раздел «История».
  2. Система запрашивает у бэкенда список завершённых броней клиента.
  3. Система отображает список прошлых посещённых классов.
- **Постусловие:** Клиент видит список своих завершённых классов (в отличие от активных будущих броней).
